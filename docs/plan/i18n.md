# Internationalization Plan

## Goal

Add a real i18n system to ask'em that can scale to all supported locales, not just English and Chinese.

The system should:

- support popup UI and injected content UI together
- support both React and non-React code paths
- allow adding new locales without rewriting product code
- preserve clear fallback behavior when translations are missing
- handle dynamic strings cleanly

This plan is for architecture and rollout. We are not implementing it yet.

## Why This Matters

ask'em now has user-visible copy spread across:

- popup React UI
- popup modals
- injected indicator / tooltip / panel UI
- shared display helpers
- legal copy

Today those strings are mostly hardcoded English. That creates three problems:

- adding one more language requires touching many unrelated files
- non-React content UI has no shared translation path
- future copy changes are easy to miss and regress

If we want true i18n, we should build for full locale expansion from day one.

Those can come later once application-level i18n is stable.

## Target Design

### 1. One shared i18n layer for the whole extension

We should not use separate translation systems for popup and content scripts.

Recommended structure:

- `src/i18n/schema.ts`
- `src/i18n/locales/en.ts`
- `src/i18n/locales/<locale>.ts`
- `src/i18n/registry.ts`
- `src/i18n/translator.ts`
- `src/i18n/react.tsx`

Core exports:

- `Locale`
- `SUPPORTED_LOCALES`
- `DEFAULT_LOCALE`
- `resolveLocale(...)`
- `createTranslator(locale)`
- `I18nProvider`
- `useI18n()`

### 2. English is the schema source of truth

The English locale file should define the message shape.

Example:

```ts
export const en = {
  popup: {
    nav: {
      home: 'Home',
      advanced: 'Advanced',
    },
  },
  content: {
    indicator: {
      ready: 'ready',
      localOnly: 'Local only',
    },
  },
} as const;

export type MessageSchema = typeof en;
```

Every other locale file should conform to `MessageSchema`.

This gives us compile-time protection against missing keys.

### 3. Support full locale expansion from the start

We should treat the system as multi-locale even if only English is fully authored on day one.

Recommended registry shape:

```ts
type LocaleMeta = {
  code: Locale;
  label: string;
  nativeLabel: string;
  rtl: boolean;
};
```

Recommended initial support list:

- `en`
- `zh-CN`
- `zh-TW`
- `ja`
- `ko`
- `fr`
- `de`
- `es`
- `pt-BR`
- `ru`

This list can expand later without changing the overall design.

### 4. Locale resolution and fallback chain

Locale choice should resolve in this order:

1. user-selected locale stored in extension settings
2. browser UI locale mapped to the nearest supported locale
3. fallback to `en`

Examples:

- `pt-BR -> pt-BR`
- `pt -> pt-BR` or `pt` if we later add it
- `zh-HK -> zh-TW`
- unknown locale -> `en`

Important rule:

- missing translation keys must fall back to English
- unsupported locales must fall back to English

### 5. Use a message formatter that works outside React

This codebase has many user-facing strings in pure utility functions, not just React components.

Because of that, the i18n layer should not be React-only.

Recommended approach:

- use ICU-style messages through `intl-messageformat`
- build a lightweight wrapper for lookup + formatting

We should not rely on a React-only library as the sole interface.

### 6. Separate semantics from copy where practical

Some current helpers return final English strings. That makes i18n harder than it needs to be.

Preferred direction:

- state helpers return semantic kinds or counts
- final copy is produced in a rendering layer through `t(...)`

Example:

Instead of:

```ts
return { label: 'current model is in sync' };
```

Prefer:

```ts
return { labelKey: 'content.indicator.currentModelInSync' };
```

or:

```ts
return { kind: 'current-model-in-sync' };
```

This does not need to be a big-bang refactor. We can phase it in.

### 7. Move long-form legal copy out of App.tsx

The legal text currently lives inline in [App.tsx](/Users/zhongxi/code/other/ask-em/src/entrypoints/popup/App.tsx).

That will become painful once we support many locales.

Recommended direction:

- move legal content into dedicated locale resources
- keep them as TS modules or markdown files under `src/i18n/legal/`

Example:

- `src/i18n/legal/en/terms.ts`
- `src/i18n/legal/en/privacy.ts`
- `src/i18n/legal/ja/terms.ts`

This keeps legal text maintainable and isolated from component logic.

## Recommended Rollout

Deliverables:

- `src/i18n/` scaffold

- locale resolver
- translator with fallback
- simple interpolation support

Files likely touched:

- [storage.ts](/Users/zhongxi/code/other/ask-em/src/runtime/storage.ts)

- `src/i18n/*`

### Phase 2: Wire locale into popup state and settings

- popup control to choose language

Files likely touched:

- `src/background/settings.ts`

### Phase 3: Migrate popup UI strings

Deliverables:

- all popup navigation, cards, controls, status labels, modals, and onboarding copy translated through the shared i18n layer
- legal copy moved out of inline component bodies

This phase should cover:

- home view
- advanced view
- feedback modal
- request-more-providers flow
- dev tools modal
- legal pages

### Phase 4: Migrate injected content UI strings

Deliverables:

- indicator labels
- sync status labels
- tooltip copy
- workspace panel copy
- provider status display labels

Priority files:

- [content-indicator.ts](/Users/zhongxi/code/other/ask-em/src/utils/content-indicator.ts)
- [workspace-provider-display.ts](/Users/zhongxi/code/other/ask-em/src/utils/workspace-provider-display.ts)

- [content-ui.ts](/Users/zhongxi/code/other/ask-em/src/utils/content-ui.ts)

### Phase 5: Clean up remaining hardcoded user-facing strings

This phase catches leftovers in:

- tests that assert visible copy
- error surfaces visible to users
- diagnostics screens
- background-originated user messages

By the end of this phase, all user-facing text should come from the i18n layer.

## Translation Resource Strategy

### Locale files

Each locale should have a dedicated module that matches the schema.

Example:

- `src/i18n/locales/en.ts`
- `src/i18n/locales/zh-CN.ts`
- `src/i18n/locales/ja.ts`

### Dynamic messages

We should not build dynamic strings through manual concatenation.

Examples that need formatter support:

- `1 of 3 synced`
- `2 models need attention`
- cooldown-related text
- counts in dev or diagnostics surfaces

Recommended:

- ICU messages via `intl-messageformat`

### Missing translation behavior

For any missing key:

- log in development
- silently fall back to English in production UI

This prevents partially translated locales from breaking the interface.

## Testing Strategy

### 1. Schema completeness

Add tests or type assertions so every locale matches the English schema.

### 2. Formatter safety

Validate that all ICU messages parse successfully.

### 3. Locale resolution

Test:

- explicit stored locale
- browser locale mapping
- unsupported locale fallback

### 4. UI regression

Add focused tests for:

- popup language switching
- key indicator states in injected UI
- representative modal text

We do not need to snapshot every string, but we should test the main rendering paths.

## Guardrails

- do not mix two i18n systems
- do not keep adding new hardcoded English strings during migration
- do not tie translation access exclusively to React
- do not concatenate user-visible copy manually when counts or variables are involved
- keep fallback behavior deterministic

## Open Questions

- Which locales do we want visible in the first public language picker?
- Do we want partially translated locales selectable, or hidden until they reach a quality threshold?
- Should legal documents ship in all locales immediately, or remain English-only until reviewed?
- Do we want a developer-only missing-key diagnostic mode?

## Recommendation

The next concrete step should be:

1. add `locale` to persistent local state
2. scaffold `src/i18n/` with an English schema and locale registry
3. wire a shared translator usable from both popup React and content utilities
4. migrate popup strings first
5. migrate injected content UI second

That keeps the first implementation tractable while still designing for full multi-locale support.
